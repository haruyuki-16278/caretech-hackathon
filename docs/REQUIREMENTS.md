# 要件定義書 — 高齢者対話・見守りSNS(仮称)

- 作成日: 2026-07-11
- ステータス: ドラフト
- 対象: caretech-hackathon 提出プロダクト

---

## 1. 背景・課題

### 1.1 課題感

1. **高齢者の気持ちの聞き取りが難しい**
   - 家族・支援者が高齢者の本音(悩み・不安・困りごと)を日常的に拾いきれない
   - 支援が必要な状態になっていても、本人から相談窓口にたどり着けない
2. **高齢者の孤独感**
   - 日常的な雑談相手の不足
   - 若い世代との接点・交流の喪失
   - 自分の経験や知恵を「伝える相手」がいない

### 1.2 目指す姿

- 高齢者が**気軽に話せる相手**(LLM)を持ち、雑談や悩みを自然に話せる
- 会話の中から**支援が必要な内容を検知**し、適切な相談窓口へ確実につなぐ
- 会話の中の**面白い話・後世に伝えたい話**を(本人の許可のもと)SNSフィードに投稿し、**若者と高齢者をつなぐハブ**になる

---

## 2. ソリューション概要

高齢者が音声で話しかけると、LLM(Azure OpenAI)が1対1で対話する。
会話内容をLLMルーターが分類し、以下の3つに振り分ける。

```
おじいちゃん/おばあちゃん
      │ 音声入力
      ▼
[録音・文字起こし] ←――――――― 担当A
      │ テキスト
      ▼
[対話LLM + ルーター] ←――――― 担当B(本書の中心スコープ)
      │
      ├─ ① 雑談 ────────── そのまま対話を継続
      │
      ├─ ② 相談(支援が必要)─ 相談エージェントへ接続
      │        └ 回答には必ず最寄り市町村の相談窓口を掲載
      │
      └─ ③ 投稿候補(面白い話・伝えたい話)
               └ 本人の許可を得たうえで SNS フィードに投稿
                        │
                        ▼
               [自作SNSフィード] ←― 若者が閲覧・リアクション
```

## 3. スコープ

### 3.1 今回作るもの(In Scope)

| # | 機能 | 概要 |
|---|------|------|
| F1 | 音声入力・文字起こし・音声出力 | 高齢者の発話をWebアプリで録音しテキスト化する。応答は音声合成して読み上げる(実装済み) |
| F2 | 対話LLM | テキストを受け取り、1対1で自然な対話応答を生成する |
| F3 | ルーター | 発話を「雑談 / 相談 / 投稿候補」に分類する |
| F4 | 相談エージェント | 支援内容の案内を生成し、**必ず**近隣市町村の相談窓口情報を付与する |
| F5 | 相談窓口DB | 対象市町村の相談窓口情報(名称・電話・受付時間等)を保持する |
| F6 | 投稿許可フロー | 投稿候補について本人に許可を確認し、許可時のみ投稿する |
| F7 | SNSフィード | 投稿を時系列で閲覧できる自作フィード(若者向け画面) |
| F8 | 画面表示 | 高齢者向けにLLM応答を大きな文字で画面表示する |

### 3.2 今回作らないもの(Out of Scope)

- 既存SNS(X / Instagram等)への外部投稿連携
- ユーザー認証・アカウント管理の本格実装(デモ用の簡易実装に留める)
- 対象市町村以外の窓口データ整備
- 緊急通報(110/119)の自動発信 ※窓口情報の表示のみ

## 4. 担当分界

| 担当 | 範囲 | 成果物 |
|------|------|--------|
| 担当A | 録音・文字起こし(F1) | 音声→テキスト変換 |
| 担当B | LLM処理(F2, F3, F4, F6)、窓口DB(F5)、フロントエンド(F7, F8)、音声入出力(F1・Webアプリ側) | FastAPI によるLLM API、UI、音声チャットページ |

**インターフェース(A↔B の境界)**: 当初は担当Aが文字起こしテキストを担当BのAPIにPOSTする想定だったが、
本実装では音声入力・音声出力(F1)を別デバイスではなく Web アプリ側(担当B)で実装した。
ブラウザで録音した音声を `/api/voice-chat` にPOSTすると、サーバー側で文字起こし→応答生成→音声合成までを行い、
応答テキスト+メタ情報(分類結果・窓口情報・投稿候補)+応答音声をJSONで返す。
テキストのみのやり取りをしたい他クライアント向けには、従来どおり `/api/chat`(テキストイン・テキストアウト)も提供する。
フィード画面・高齢者向け画面の表示も担当Bが実装する。

## 5. 機能要件

### 5.1 対話LLM(F2)

- Azure OpenAI のチャットモデルを使用する
- 高齢者に寄り添う口調(丁寧・ゆっくり・共感的)をシステムプロンプトで規定する
- 会話履歴(セッション単位)を保持し、文脈を踏まえた応答を返す

> **実装状況(2026-07-11 追記)**: システムプロンプトはカテゴリ(`chat` / `consult` / `story`)ごとに
> 切り替えており、`story` では応答の最後に必ず投稿可否を確認する一文を含めるよう明示的に指示している
> (`app/services/chat.py`)。会話履歴は `session_id` ごとにプロセス内メモリで直近6ターン分のみ保持し
> (`app/services/session_store.py`)、Azure OpenAI 呼び出し時にシステムプロンプトに続けて渡す簡易実装。
> 永続化やマルチプロセス共有には対応していない。

### 5.2 ルーター(F3)

- 各発話を以下のいずれかに分類する
  - `chat` — 雑談。対話を継続
  - `consult` — 悩み・困りごと・支援が必要な可能性のある内容
  - `story` — 面白い話・後世に伝えたい話(投稿候補)
- 分類はLLMによる判定(structured output / function calling)で行う
- 1発話が複数に該当する場合は `consult` を最優先とする(安全側に倒す)

> **実装状況(2026-07-11 追記)**: 現状はLLMではなく**キーワードベースの簡易ルーター**
> (`app/services/router.py`)で実装している(ハッカソン規模のため)。「複数該当時は consult を
> 最優先」という安全側の要件は満たすが、キーワードに一致しない発話は文脈的に気がかりな内容でも
> `chat` に分類される、逆に過去の出来事を振り返っただけの発話でもキーワードが含まれれば
> `consult` に分類されるといった誤検知が起こり得る(`tests/test_router.py` に既知の挙動として明記)。
> LLMベースの判定への置き換えは今後の課題とする。

### 5.3 相談エージェント(F4)

- `consult` に分類された発話に対し、支援情報の案内を生成する
- **応答には必ず、利用者の居住地に最も近い市町村の相談窓口を掲載する**(必須要件)
- 窓口情報は窓口DB(F5)から取得し、LLMの生成テキストに埋め込むのではなく構造化データとして返す(ハルシネーション防止)
- 医療・法律に関わる断定的な助言はせず、窓口への相談を促すトーンとする

### 5.4 相談窓口DB(F5)

- **対象地域: 坂井地区広域連合(福井県あわら市・坂井市)**
- 高齢者の総合相談窓口である**地域包括支援センター**(あわら / 三国 / 丸岡 / 春江 / 坂井 の5拠点)を初期データとして登録済み → [`data/consultation_offices.json`](../data/consultation_offices.json)
- 利用者の居住エリア(市・地区)に応じて最寄りのセンターを返す
- データ項目: 市町村名 / エリア / 窓口名 / 分野 / 住所 / 電話番号 / FAX
- 形式: JSON(ハッカソン規模のため軽量に)。出典は坂井地区広域連合の介護保険パンフレット。デモ前に公式サイトで最新情報を確認する

> **実装状況(2026-07-11 追記)**: 「福井県あわら市」のように都道府県名が前置されていたり、
> 「福井県坂井市丸岡町」のように住所文字列の一部として地区名が埋め込まれていても、
> できるだけ具体的な窓口を特定するように検索ロジックを強化した。一方で「坂井市」のように
> 1つの市町村に複数の地域包括支援センターが存在する場合は、地区を特定できないまま1件を
> 機械的に選ぶのではなく、広域窓口(坂井地区広域連合)を返しつつ、候補地区名を
> `consult_info.note` に明示する(`app/services/offices.py`)。

### 5.5 投稿許可フロー(F6)

- `story` に分類された場合、LLMが投稿文案を生成し「SNSに投稿してもいいですか?」と本人に確認する
- 明示的な同意(はい等)が得られた場合のみ投稿する
- 拒否・無応答の場合は投稿せず、雑談として継続する

> **実装状況(2026-07-11 追記)**: `session_id` ごとに投稿確認待ち状態(`pending_story`)を保持し、
> 次のターンで「はい/うん/お願いします」等の肯定表現、または「いいえ/やめて」等の否定表現を
> 簡易キーワードで判定する最小実装を追加した(`app/services/conversation.py`)。
> 判定できない返答の場合は確認待ち状態を維持し、通常発話として処理を継続する
> (会話の脱線に対する明示的なキャンセル操作は未実装)。実際の投稿ストア(F7連携)への
> 保存はまだ行わず、レスポンスの `posted` / `awaiting_confirmation` フラグで確認状況を表現するに留める。

### 5.6 SNSフィード(F7)

- 投稿を新しい順に表示する自作フィード
- 投稿には投稿者の表示名(ニックネーム)と本文を表示
- 若者側からのリアクション(いいね等)はストレッチゴール

## 6. 非機能要件

| 項目 | 要件 |
|------|------|
| 応答速度 | 発話からLLM応答表示まで実用的な速度(目安: 数秒以内) |
| 可用性 | ハッカソンデモが成立すれば十分(SLA不要) |
| プライバシー | 会話内容はデモ用途に限定。本人許可のない内容はフィードに出さない |
| 安全性 | 相談分類を最優先し、支援が必要な発話の取りこぼしを防ぐ |
| アクセシビリティ | 高齢者向け画面は大きな文字・シンプルな構成 |
| 秘密情報 | Azure OpenAI のAPIキーは環境変数で管理し、リポジトリにコミットしない |

## 7. 技術スタック

| レイヤ | 技術 | 担当 |
|--------|------|------|
| 音声入力・文字起こし | ブラウザ(`MediaRecorder` / `getUserMedia`)で録音し `/api/voice-chat` に送信。サーバー側は Azure Speech to Text(環境変数未設定時はフォールバック応答) | B(当初は担当A予定だったが、Webアプリ側で対応) |
| 音声出力(TTS) | サーバー側 Azure Speech(SSML)で音声合成し base64 で返却。未設定時はブラウザの `SpeechSynthesis` (Web Speech API) にフォールバック | B |
| フロントエンド | FastAPI + Jinja2 テンプレートで配信するシンプルな HTML/CSS/JS(高齢者向け大きな文字・ボタン) | B |
| LLM API | Python + FastAPI | B |
| LLM | Azure OpenAI(チャットモデル。未設定時はルールベースの簡易応答にフォールバック) | B |
| 窓口データ | JSON(`data/consultation_offices.json`) | B |
| 投稿ストア | SQLite 等の軽量ストア(未実装・今後対応) | B |

> **補足(2026-07-11 更新)**: 当初「担当A」が受け持つ想定だった音声入力・音声出力(F1)は、
> 別デバイスではなく Web アプリ(ブラウザ)側で実装する方針に変更した。
> 詳細は [`README.md`](../README.md) の「音声チャットページ」を参照。

## 8. API インターフェース(担当A↔B 契約・ドラフト)

### POST `/api/chat`

高齢者の発話テキストを送り、応答と分類結果を受け取る。

リクエスト:

```json
{
  "session_id": "uuid",
  "user_id": "demo-user-1",
  "text": "最近、腰が痛くて買い物に行くのがつらいんだよ"
}
```

レスポンス:

```json
{
  "reply": "それはおつらいですね。……お住まいの地域に相談できる窓口がありますよ。",
  "category": "consult",
  "consult_info": {
    "municipality": "坂井市",
    "area": "丸岡",
    "office": "丸岡地域包括支援センター",
    "field": "高齢者総合相談(介護・福祉・介護予防・権利擁護)",
    "address": "坂井市丸岡町西瓜屋15-12",
    "phone": "0776-68-1130",
    "note": null
  },
  "post_proposal": null,
  "awaiting_confirmation": false,
  "posted": false
}
```

- `consult_info.note` には、市町村名だけでは地区(窓口)を一意に特定できない場合
  (例:「坂井市」のみ指定され、三国・丸岡・春江・坂井の複数窓口が該当する場合)に、
  候補となる地区名を明示する補足メッセージが入る(通常は `null`)。
- `awaiting_confirmation` は、`story` の投稿可否をユーザーに確認中で、次発話が
  「はい/いいえ」等の返答として扱われるかどうかを示す。
- `posted` は、この応答内でユーザーの同意により投稿(story)が確定したかどうかを示す。
  `category: "story"` の初回応答では `post_proposal` に投稿文案が入り
  `awaiting_confirmation: true` となる。続くターンで同意(「はい」等)が確認されると、
  `posted: true` となり `post_proposal` には確定した投稿内容が入る(実際の投稿ストアへの
  保存は未実装。フィード(F7)実装時に連携する)。

### POST `/api/voice-chat`(音声入力・音声出力対応。本実装で追加)

ブラウザで録音した音声ファイルを `multipart/form-data` で受け取り、音声認識(STT)→ 対話LLM/ルーター/窓口検索 → 音声合成(TTS)までをまとめて処理する。

リクエスト(`multipart/form-data`):

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `audio` | ファイル | ○ | 録音された音声(webm/ogg/wav 等)。上限サイズ(既定10MB、`VOICE_MAX_UPLOAD_BYTES` で変更可)を超える場合は `413` を返す |
| `session_id` | string | - | セッションID |
| `user_id` | string | - | ユーザーID |
| `area` | string | - | 利用者の居住エリア(相談窓口検索に使用) |

レスポンス:

```json
{
  "transcript": "最近、腰が痛くて買い物に行くのがつらいんだよ",
  "reply": "それはおつらいですね。……\n\nお近くの相談窓口: 丸岡地域包括支援センター(...) 電話: 0776-68-1130",
  "category": "consult",
  "consult_info": { "municipality": "坂井市", "area": "丸岡", "office": "丸岡地域包括支援センター", "field": "...", "address": "...", "phone": "0776-68-1130", "note": null },
  "post_proposal": null,
  "audio_base64": "<base64エンコードされたmp3>",
  "audio_content_type": "audio/mpeg",
  "tts_available": true,
  "stt_available": true,
  "awaiting_confirmation": false,
  "posted": false
}
```

- `AZURE_SPEECH_KEY` / `AZURE_SPEECH_REGION` が未設定、または音声認識に失敗した場合は `transcript` が `null` になり、聞き取れなかった旨の `reply` を返す(`audio_base64` も `tts_available` に応じて `null` になりうる)。
- `audio_base64` が `null` の場合、フロントエンドはブラウザ標準の音声合成(Web Speech API)で読み上げる(サーバー側TTS未設定でもページは利用可能)。
- `awaiting_confirmation` / `posted` の意味は `/api/chat` と同様(F6: 投稿許可フロー)。

### GET `/api/posts`(未実装・今後対応)

SNSフィード用の投稿一覧を返す(新しい順)。

> 詳細スキーマは実装開始時に `docs/API.md` として分離・確定する。

## 9. デモシナリオ(想定)

1. 高齢者が雑談を話しかける → LLMが自然に応答(`chat`)
2. 「腰が痛くて買い物がつらい」と話す → 相談エージェントが共感+**市町村窓口を表示**(`consult`)
3. 「昔、こんな面白いことがあってなぁ」と話す → LLMが投稿を提案 → 本人が「いいよ」と許可 → フィードに投稿(`story`)
4. 若者側画面でフィードを開き、投稿が見えることを確認
